# 实践 1.1 实验报告：学习 Flex，用 Flex 实现词法分析

## 一、目的和任务

本次实验的目的在于学习 Flex 的基本使用方法，理解词法分析器的工作流程，并使用 Flex 实现一个面向 C-- 语言的词法分析程序。通过本次实验，需要掌握如何用正则表达式描述词法规则，如何让 Flex 根据规则生成扫描器，以及如何在扫描过程中识别词法单元、记录行号并报告词法错误。

本次实验的主要任务是编写 `lexer.l` 文件，实现 C-- 源程序的词法分析。程序需要能够从命令行接收一个源文件名作为输入，逐行扫描源代码，识别关键字、标识符、整数、浮点数、运算符、界符和注释等词法成分，并按照实验要求输出相应的词法单元。对于源程序中出现的非法字符或非法词法形式，程序需要输出错误类型 A、错误所在行号以及简要说明。

本次实现不仅完成了基本词法单元识别，还加入了部分扩展要求，包括八进制整数、十六进制整数、指数形式浮点数、`//` 行注释和 `/* ... */` 块注释的识别与处理。当前实验只完成词法分析部分，尚未接入 Bison，因此不生成语法树，也不处理语法错误类型 B。

## 二、过程

首先根据实验要求整理 C-- 语言中需要识别的词法单元。C-- 程序中的基本词法成分包括关键字、标识符、常量、运算符、界符和注释。其中关键字包括 `int`、`float`、`struct`、`return`、`if`、`else`、`while`；标识符由字母、下划线和数字组成，但不能以数字开头；常量包括整数常量和浮点常量；运算符包括赋值运算符、关系运算符、算术运算符和逻辑运算符；界符包括括号、方括号、花括号、分号、逗号和点号。

随后创建实践目录 `exp1.1`，并在其中编写 Flex 源文件 `lexer.l`。文件开头使用：

```lex
%option noyywrap yylineno
```

其中 `noyywrap` 用于避免额外定义 `yywrap` 函数，`yylineno` 用于让 Flex 自动维护当前行号，方便在出现词法错误时输出错误位置。

在 C 代码段中定义了几个辅助函数。`print_token` 用于输出普通词法单元，`print_text_token` 用于输出带词素内容的词法单元，例如 `ID: main` 和 `TYPE: int`。`print_int_token` 使用 `strtol` 将十进制、八进制和十六进制整数字面量统一转换为十进制输出。`print_float_token` 使用 `strtod` 将浮点数字面量转换为浮点值输出。`report_error` 用于统一输出词法错误，格式为：

```text
Error type A at Line [行号]: [说明文字].
```

接着使用 Flex 正则定义常用模式。例如：

```lex
DIGIT       [0-9]
LETTER      [_A-Za-z]
ID          {LETTER}({LETTER}|{DIGIT})*
DECINT      0|[1-9]{DIGIT}*
OCTINT      0[0-7]+
HEXINT      0[xX][0-9A-Fa-f]+
```

其中 `ID` 用于匹配标识符，`DECINT`、`OCTINT`、`HEXINT` 分别用于匹配十进制、八进制和十六进制整数。为了能够直接发现一些常见词法错误，还额外定义了 `BAD_OCT`、`BAD_HEX`、`BAD_FLOAT1`、`BAD_FLOAT2` 等模式，用于识别非法八进制数、非法十六进制数和非法指数浮点数。

在规则部分中，先处理注释和空白字符。对于 `//` 行注释，直接匹配并忽略整行内容。对于 `/* ... */` 块注释，使用 Flex 的开始条件 `%x COMMENT` 进入注释状态，直到读到 `*/` 后回到普通状态。如果文件结束时仍处在注释状态，则说明块注释未闭合，此时输出词法错误 A。

之后依次处理关键字、常量、标识符、运算符和界符。关键字规则放在标识符规则之前，这样 `int`、`float`、`if` 等词素会优先被识别为关键字，而不会被错误识别为普通标识符。整数和浮点数的规则也按照非法形式优先、合法形式随后处理的顺序书写，使得 `09`、`0x3G`、`1.05e` 等输入可以直接报告为词法错误，而不是被拆分成多个合法词法单元。

最后添加兜底规则：

```lex
.  { report_error("Mysterious character", yytext); }
```

这条规则用于捕获所有未被前面规则识别的字符，例如 `~`。当扫描器读到这种字符时，会按照错误类型 A 输出错误信息。

为了方便编译和测试，本次实验还编写了 `Makefile`。在 `exp1.1` 目录下执行：

```bash
make clean
make
```

即可先由 Flex 生成 `lex.yy.c`，再调用 C 编译器生成可执行文件 `lexer`。运行方式为：

```bash
./lexer tests/lex_ok.cmm
./lexer tests/lex_error.cmm
```

其中 `tests/lex_ok.cmm` 用于测试合法词法单元，包括注释、八进制整数、十六进制整数、指数浮点数、关系运算符和关键字；`tests/lex_error.cmm` 用于测试词法错误，包括非法八进制数 `09`、非法十六进制数 `0x3G`、非法指数浮点数 `1.05e` 和未定义字符 `~`。

## 三、结果与结论分析

通过本次实验，已经完成了一个可以独立运行的 Flex 词法分析器。当前目录中的主要文件包括：

```text
lexer.l
Makefile
tests/lex_ok.cmm
tests/lex_error.cmm
README.md
```

执行 `make test` 后，合法样例可以被正常扫描，程序能够输出 `TYPE`、`ID`、`INT`、`FLOAT`、`IF`、`RETURN`、`ELSE`、`RELOP`、`ASSIGNOP`、`SEMI`、`LP`、`RP`、`LC`、`RC` 等词法单元。对于 `0123`，程序输出十进制整数值 `83`；对于 `0x3F`，程序输出十进制整数值 `63`；对于 `1.05e-4`，程序输出浮点值 `0.000105`，说明扩展数字形式的识别和数值转换能够正常工作。

错误样例也能够按照实验要求报告词法错误。例如，程序可以输出非法八进制数、非法十六进制数、非法浮点数和未定义字符对应的错误类型 A，并能够给出正确的行号。错误输出形式符合：

```text
Error type A at Line [行号]: [说明文字].
```

本次实验说明，Flex 可以将词法规则从手写扫描逻辑中抽离出来，只需要用正则表达式描述不同词法单元的模式，再在匹配动作中完成输出、转换和错误处理即可。通过规则顺序的控制，可以解决关键字与标识符、合法数字与非法数字之间的优先级问题。通过开始条件机制，可以比较清晰地处理块注释这类跨多行结构。

目前程序已经完成实践 1.1 所需的词法分析功能，但它仍然是一个独立扫描器，没有与 Bison 语法分析器连接。因此，当输入中存在语法错误时，本程序不会报告错误类型 B，也不会构造或打印语法树。后续实验可以在当前词法分析器的基础上，将词法单元返回给 Bison，并进一步完成语法分析和语法树构造。
