# 编译原理课程设计环境配置记录

## 1. 背景

本次配置的目标是在自己的 Mac 上搭建编译原理课程设计相关环境。课堂给出的参考环境为：

- GNU/Linux：Ubuntu 20.04
- GCC：7.5.0
- GNU Flex：2.6.4
- GNU Bison：3.5.1
- QtSpim：9.1.9
- 课程提供的虚拟机小程序 irsim

最开始我以为所有内容都必须完全依赖 Linux 环境，后来梳理后发现，编译原理实验可以分成两部分：

- 用 GCC、Flex、Bison 等工具编写和编译自己的编译器。
- 用 QtSpim 或课程提供的 irsim/interpret 运行编译器生成的目标代码或中间代码。

因此，Mac 上可以完成大部分开发和验证工作，但如果要和老师机房环境完全一致，Ubuntu 20.04 虚拟机会更稳。

## 3. QtSpim 的理解与安装

一开始容易把 QtSpim 误解成 Qt 开发环境。实际上，QtSpim 是 SPIM 的图形版 MIPS32 汇编模拟器。

QtSpim 不运行 `.exe`，也不运行已经编译好的二进制程序。它读取并执行 MIPS 汇编文本，例如 `.s` 或 `.asm` 文件。

它在课程设计中的作用更像是：

```text
源程序
  ↓
编译器
  ↓
生成 MIPS32 汇编
  ↓
QtSpim 加载并运行汇编程序
```

本机已经安装 QtSpim，验证到的位置为：

```text
/Applications/QtSpim.app
```

检测到的版本为：

```text
QtSpim 9.1.23
```

虽然课堂截图中是 9.1.9 或 9.1.19，但目前这个版本可以用于基本 MIPS 汇编程序验证。

```text

```

这说明 QtSpim 本身可以正常加载和运行 MIPS 汇编程序。

## 5. 课程项目与 QtSpim 的关系

检查当前课程项目后发现，目录里没有现成的 `.s` 或 `.asm` 文件。因此这个项目目前并不是直接交给 QtSpim 跑的。

`PL.cpp` 中的输出函数显示，它会生成：

- `.lst`：可读的代码列表
- `.pld`：二进制形式的课程虚拟机代码
- `.lab`：标签/跳转地址信息

所以当前项目更像是：

```text
课程源程序
  ↓
PL 编译器
  ↓
生成 .pld / .lst / .lab
  ↓
interpret 或 irsim 解释执行 .pld
```

这和 QtSpim 的 MIPS 汇编运行流程不是同一条链路。

## 6. 在 Mac 上编译课程代码遇到的问题

尝试直接编译 `PL.cpp` 和 `interpret.cpp` 时，遇到了老 Windows 工程迁移到 macOS 的兼容问题。

### 6.1 conio.h 缺失

错误示例：

```text
fatal error: 'conio.h' file not found
```

原因是 `conio.h` 是 Windows/DOS 风格的头文件，macOS 上没有。

为了验证解释器部分，我临时创建了一个兼容头文件：

```text
/Users/shengjianuo/Desktop/compiler/conio.h
```

其中只简单适配了 `getch()`：

```c
#pragma once

#include <stdio.h>

static inline int getch(void)
{
    return getchar();
}
```

### 6.2 interpret.cpp 的老式 C++ 写法

`interpret` 部分还遇到两个现代编译器不接受的问题：

```cpp
const STACKSIZE=2047,CXMAX=10000,LEVMAX=100;
```

需要改成：

```cpp
const int STACKSIZE=2047,CXMAX=10000,LEVMAX=100;
```

另外：

```cpp
void main(int arg,char ** argv)
```

需要改成：

```cpp
int main(int arg,char ** argv)
```

修改后，`interpret` 已经可以在 Mac 上编译。

编译命令为：

```bash
cd /Users/shengjianuo/Desktop/compiler
g++ -std=gnu++98 -Wno-invalid-source-encoding -I. C/src/interpret/interpret.cpp -o interpret
```

生成文件：

```text
/Users/shengjianuo/Desktop/compiler/interpret
```

运行不存在的 `.pld` 文件时，程序能正常启动并提示打不开文件，说明可执行程序本身已经可用：

```bash
./interpret /tmp/nonexistent.pld
```

### 6.3 PL.cpp 的 MFC 依赖

`PL.cpp` 暂时还不能直接在 Mac 上编译，主要问题不只是 `conio.h`，还包括：

```cpp
#include <Afxtempl.h>
class SYMLIST: public CList<SYMBOL,SYMBOL>
```

`Afxtempl.h` 和 `CList` 是 Microsoft Visual C++ / MFC 相关内容，macOS 上没有。

因此，`PL.cpp` 需要进一步移植，例如把 `CList` 替换为标准 C++ 容器，如 `std::list` 或 `std::vector`。这部分改动比 `interpret` 大，不能只靠编译参数解决。

目前已经完成的验证：

- QtSpim 已安装，位置为 `/Applications/QtSpim.app`。

## 8. 后续计划

后续可以按下面顺序继续：

1. 准备一个课程语言的测试源程序。
2. 移植 `PL.cpp`，主要处理 `Afxtempl.h`、`CList`、`stricmp` 等 Windows 依赖。
3. 在 Mac 上编译出 `PL`。
4. 用 `PL` 编译测试源程序，生成 `.pld`、`.lst`、`.lab`。
5. 用 `interpret` 或 `irsim` 运行 `.pld`。
6. 如果课程后续要求生成 MIPS 汇编，再用 QtSpim 验证 `.s` 或 `.asm` 文件。

总体结论是：Mac 上可以搭建和验证大部分编译原理实验环境，但当前课程源码来自老 Visual C++ 工程，完整迁移到 Mac 需要做一定的跨平台适配。如果只是为了和老师环境完全一致，Ubuntu 20.04 虚拟机仍然是更稳的选择。
