# 实践二实验报告：语义分析与符号表

## 一、目的与任务

本次实验的目标是在实践一完成的词法分析和语法分析基础上，将原来的语法分析器继续改造为翻译器前端的一部分，即语义分析器。程序需要在语法正确的 C-- 源程序上建立符号表和类型信息，并检查实验要求中列出的 17 类语义错误。

本实验的任务重点有三项。第一，设计符号表结构，能够记录变量、函数和结构体等不同名字空间中的实体。第二，设计类型系统，能够表达基本类型、数组类型、结构体类型和函数类型，并支持类型等价判断。第三，在遍历语法树的过程中完成语义检查，包括未定义、重复定义、类型不匹配、非法操作符使用、函数调用参数不匹配、结构体成员访问错误等。

本次实现采用实践二的必做假设：所有变量和形参位于全局作用域，变量不能重名；函数只能定义不能声明；结构体采用名等价；函数不能嵌套定义。选做要求中的函数声明、嵌套作用域和结构等价暂未实现，但当前接口为后续扩展保留了空间。

## 二、过程

### 1. 复用语法前端

实验二继续复用实践 1.2 中的 Flex/Bison 前端。`lexer.l` 负责识别 token，`parser.y` 负责构造 AST。与实践 1.2 不同的是，实验二不再输出语法树，而是在语法分析成功后调用 `semantic_analyze()` 进入语义分析阶段。如果输入没有语义错误，程序不输出任何内容；如果存在语义错误，则输出错误信息。

这种设计保持了阶段之间的黑盒接口：语义分析只接收 AST，不依赖词法分析正则表达式或 Bison 规约动作的内部细节。

### 2. 符号表设计

语义分析中维护三张链式符号表：

- 变量表：保存普通变量和函数形参。根据必做假设，所有变量共享全局作用域，所以局部变量和形参也需要检查全局重名。
- 函数表：保存函数名、返回类型、形参数量和形参类型链表，用于检查函数重复定义、未定义函数和调用参数匹配。
- 结构体表：保存结构体名及其字段链表，用于检查结构体重复定义、未定义结构体和字段访问。

符号表使用简单链表实现，而不是套用模板化哈希表。这样做虽然查找复杂度较高，但实验规模较小，链表结构更透明，也便于在报告和后续调试中说明每一类符号的来源和生命周期。

### 3. 类型系统设计

类型系统使用 `Type` 结构表示五类类型：

- `TYPE_BASIC`：基本类型，包括 `int` 和 `float`。
- `TYPE_ARRAY`：数组类型，保存元素类型和数组大小。
- `TYPE_STRUCT`：结构体类型，保存结构体名和字段链表。
- `TYPE_FUNCTION`：函数类型，保存返回类型和形参链表。
- `TYPE_ERROR`：错误类型，用于抑制连锁错误。

类型等价规则按必做要求实现。基本类型要求完全相同；数组类型只比较基类型和维数结构，不要求每一维大小相同；结构体采用名等价，即结构体名相同才认为类型相同。匿名结构体通过独立类型节点表示，因此不会被误判为同名类型。

### 4. 语义检查流程

语义分析从 `Program` 节点开始递归遍历 AST。遇到外部定义时，先分析 `Specifier` 得到基础类型，再根据后续节点判断是全局变量、结构体定义还是函数定义。

变量定义会进入变量表，并检查是否与已有变量或结构体同名。函数定义会进入函数表，并记录返回类型和形参类型；之后分析函数体中的局部定义和语句。结构体定义会进入结构体表，并分析字段定义，检查字段重复定义和字段初始化。

表达式分析函数会返回表达式类型。赋值表达式检查左侧是否为左值，并比较左右类型；算术和逻辑表达式检查操作数类型；数组访问检查被访问对象是否为数组、下标是否为整数；结构体成员访问检查对象是否为结构体以及字段是否存在；函数调用检查函数是否定义、是否误把变量当函数、实参与形参数量和类型是否匹配。

### 5. 错误处理

错误信息统一通过 `semantic_error()` 输出，格式为：

```text
Error type [错误类型] at Line [行号]: [说明文字].
```

当某个表达式已经产生本质错误时，其类型会被标记为 `TYPE_ERROR`。后续表达式如果使用这个错误类型，会尽量避免继续产生无意义的连锁类型错误。例如结构体字段不存在时，成员访问表达式返回错误类型，外层关系运算不会再额外报告操作数类型不匹配。

## 三、结果与分析

### 1. 编译运行

实验目录提供 Makefile：

```bash
cd ~/compiler/exp2.0
make clean
make
```

运行全部样例：

```bash
make test
```

构建过程使用 Bison 生成 `parser.tab.c/parser.tab.h`，使用 Flex 生成 `lex.yy.c`，再与 `ast.c`、`semantic.c` 一起编译为可执行程序 `cc`。

### 2. 测试结果

`test/00_ok.cmm` 是无语义错误样例，运行后不输出任何内容，符合实验二要求。

`test/01_undefined_variable.cmm` 到 `test/17_undefined_struct.cmm` 分别覆盖错误类型 1 至 17。当前 `make test` 能输出对应错误类型，例如：

```text
Error type 1 at Line 4: Undefined variable "j".
Error type 4 at Line 6: Redefined function "func".
Error type 9 at Line 8: Function arguments mismatched for "func".
Error type 17 at Line 3: Undefined structure "Position".
```

这说明符号表能够记录并区分变量、函数、结构体，表达式类型推导也能支持后续语义判断。

### 3. 分析与后续扩展

本实现的关键点是把语义分析做成独立阶段。语义分析不读取 token 文本流，也不依赖语法树的打印结果，而是直接使用 AST 和符号表接口。这种设计可以继续向后连接中间代码生成：只要语义分析通过，后续 IR 生成器就可以复用 AST、变量类型和函数签名信息。

当前实现采用全局变量作用域和结构体名等价，符合必做部分要求。后续若实现选做要求，可以在现有结构上扩展：函数声明可以在函数表中增加“已声明/已定义”状态；嵌套作用域可以将变量表改造成作用域栈；结构等价可以替换 `same_type()` 中的结构体比较逻辑。
