# 实践 1.2 实验报告：基于 Bison 的语法分析

## 一、目的与任务

本次实验的目标是在已经完成的 Flex 词法分析基础上，继续使用 Bison 实现 C-- 语言的语法分析部分，使程序能够从源代码文件中识别合法的语法结构，构造语法树，并按照课程要求输出分析结果。

实验需要完成的核心任务包括三类。第一，程序应能够接收一个 C-- 源文件路径作为命令行参数，并以该文件作为输入进行分析。第二，程序需要识别词法错误和语法错误：词法错误属于错误类型 A，例如未定义字符、非法八进制数、非法十六进制数、非法指数浮点数和未闭合注释；语法错误属于错误类型 B，例如语句不符合 C-- 文法、数组访问格式错误或语句缺少分号。第三，当输入文件没有任何词法或语法错误时，程序应构造语法树，并按照先序遍历的方式输出节点信息；当输入文件存在错误时，程序只输出错误信息，不输出语法树。

除必做内容外，本实验也继续支持实践 1.1 中已经完成的选做词法功能，包括八进制和十六进制整数识别、指数形式浮点数识别，以及 `//` 行注释和 `/* ... */` 块注释过滤。这样，语法分析阶段不仅可以处理基本 C-- 程序，也可以正确接收这些扩展词法单元。

从整体编译器设计角度看，本次实验还承担一个接口衔接任务。词法分析不应再仅仅把 token 打印到标准输出，而应把 token 类型、词素内容、数值和行号以结构化方式传递给语法分析器。语法分析器也不应只为了当前输出格式临时拼接字符串，而应生成后续语义分析、符号表构建和中间代码生成可以继续使用的抽象语法树。

## 二、过程

### 1. 调整词法分析接口

实践 1.1 中的 `lexer.l` 主要用于单独验证词法分析能力，因此每识别一个 token 就直接打印结果。这种方式适合词法实验，但不适合作为语法分析器的输入接口。为了解决这个问题，本次实验在 `exp1.2/lexer.l` 中重新组织词法动作，使 Flex 在识别 token 后不再直接打印，而是返回 Bison 定义的 token 类型。

对于需要携带语义信息的 token，词法分析器会创建一个 AST 词法节点并写入 `yylval.node`。例如，`ID` 节点保存标识符名字，`TYPE` 节点保存 `int` 或 `float`，`INT` 和 `FLOAT` 节点保存转换后的十进制数值文本。八进制整数和十六进制整数通过 `strtol` 转换为十进制，指数形式浮点数通过 `strtod` 转换为普通浮点数输出格式。这样，Bison 规约时可以直接拿到结构化节点，不需要从文本输出中反向解析。

词法错误仍然在词法阶段报告。非法数字、未定义字符和未闭合块注释会输出 `Error type A at Line ...`。如果分析过程中出现词法错误，程序会继续扫描后续输入，尽量报告文件中的多个词法错误；同时语法阶段会抑制由坏 token 连带造成的语法错误噪声，避免同一个根本问题重复报错。

### 2. 使用 Bison 描述 C-- 文法

语法分析部分位于 `exp1.2/parser.y`。该文件使用 Bison 描述 C-- 的主要文法结构，包括：

- 程序入口：`Program`、`ExtDefList`
- 外部定义：全局变量声明、结构体定义、函数定义
- 类型说明：基本类型和结构体类型
- 声明部分：变量、数组、函数参数、局部定义
- 语句部分：复合语句、表达式语句、返回语句、条件语句、循环语句
- 表达式部分：赋值、逻辑运算、关系运算、算术运算、括号表达式、函数调用、数组访问、结构体成员访问

表达式文法中设置了优先级和结合性，例如赋值运算右结合，逻辑或低于逻辑与，关系运算低于加减乘除，一元负号和逻辑非具有较高优先级。`if-else` 的悬挂 else 问题通过 Bison 的优先级声明处理，使 `else` 与最近的未匹配 `if` 结合。

### 3. 构造语法树

为了降低代码雷同风险并方便后续扩展，本实验没有采用固定数组式的模板 AST，而是设计了一个轻量级链式 AST 结构，定义在 `exp1.2/ast.h` 中。每个节点包含节点名、可选文本、行号、节点类型标记以及子节点链表指针：

```c
struct ASTNode {
    char *name;
    char *text;
    int line;
    int is_token;
    ASTNode *first_child;
    ASTNode *last_child;
    ASTNode *next_sibling;
};
```

词法节点和语法节点共用这一结构。词法节点用于保存 `ID`、`TYPE`、`INT`、`FLOAT` 等 token 信息；语法节点用于保存 `Program`、`ExtDef`、`Stmt`、`Exp` 等非终结符。Bison 的每个规约动作通过 `ast_branch()` 创建父节点，并按产生式顺序挂接子节点。这样做可以保证打印结果天然符合先序遍历，也让后续语义分析能够按树结构递归处理。

空产生式不创建节点，因此打印时不会出现课程要求中不应输出的 ε 节点。语法节点的行号取其第一个有效子节点的行号，符合“语法单元行号为其产生出的所有词素中第一个出现位置”的要求。

### 4. 错误处理与恢复

本实验对错误进行了分类处理。词法错误由 Flex 发现并输出错误类型 A；语法错误由 Bison 的错误恢复规则输出错误类型 B。

对于通用语法错误，程序可以输出 `Syntax error`。除此之外，本实现针对课程样例中常见的错误做了更明确的恢复：

- `a[5,3]` 会报告 `Missing "]"`，因为二维数组访问应写成 `a[5][3]`；
- `if (a[1][2] == 0) i = 1 else i = 0;` 会报告 `Missing ";"`，因为 `else` 前的赋值语句缺少分号；
- 局部定义或外部定义缺少分号时，也会报告 `Missing ";"`。

为了满足“有错误时不输出语法树”的要求，主程序在 `yyparse()` 结束后检查词法错误计数和语法错误计数。只有两者都为 0 且解析成功时，才调用 `ast_print()` 输出语法树。

### 5. 测试样例组织

课程文档中的必做样例和选做样例已经单独放在 `exp1.2/tests/` 目录下。必做样例文件命名为 `required_*.cmm`，选做样例文件命名为 `optional_*.cmm`。

必做样例包括：

- `required_1_lex_error.cmm`：未定义字符 `~`
- `required_2_syntax_error.cmm`：数组访问逗号错误和缺少分号
- `required_3_tree_inc.cmm`：简单函数语法树输出
- `required_4_tree_struct.cmm`：结构体定义和成员访问语法树输出

选做样例包括：

- `optional_1_base_number_ok.cmm`：合法八进制和十六进制整数
- `optional_2_base_number_error.cmm`：非法八进制和十六进制整数
- `optional_3_float_exp_ok.cmm`：合法指数形式浮点数
- `optional_4_float_exp_error.cmm`：非法指数形式浮点数
- `optional_5_comment_ok.cmm`：合法行注释和块注释
- `optional_6_nested_comment_error.cmm`：嵌套注释导致的语法错误

可以使用以下命令一次运行这些样例：

```bash
cd ~/compiler/exp1.2
make sample-test
```

## 三、结果与分析

### 1. 编译结果

实验目录提供 Makefile，可以在 Ubuntu 20.04、GCC 7.5.0、Flex 2.6.4、Bison 3.5.1 环境下直接构建：

```bash
cd ~/compiler/exp1.2
make clean
make
```

构建过程会先由 Bison 根据 `parser.y` 生成 `parser.tab.c` 和 `parser.tab.h`，再由 Flex 根据 `lexer.l` 生成 `lex.yy.c`，最后使用 GCC 将生成文件和 `ast.c` 一起编译为可执行程序 `cc`。

Flex 生成的 `lex.yy.c` 在 `-Wall -Wextra -std=c99` 下会出现少量常见 warning，例如 `fileno` 的隐式声明以及 `input`、`yyunput` 未使用。这些 warning 来自 Flex 生成代码，不影响程序运行。项目自身代码避免使用 glib 等额外库，只依赖 C 标准库、Flex 和 Bison。

### 2. 运行结果

运行综合测试：

```bash
make test
```

其中 `syntax_ok.cmm` 会输出完整语法树，说明程序能够识别结构体定义、函数定义、局部变量定义、赋值表达式、结构体成员访问、十六进制整数和指数浮点数等结构。

`syntax_error.cmm` 的输出为：

```text
Error type B at Line 5: Missing "]".
Error type B at Line 6: Missing ";".
```

这说明语法分析器能够在同一文件中报告多处语法错误，并且可以针对课程样例给出较明确的错误说明。

`lex_error.cmm` 的输出为：

```text
Error type A at Line 3: Illegal octal number "09".
Error type A at Line 4: Illegal hexadecimal number "0x3G".
Error type A at Line 5: Illegal floating point number "1.05e".
Error type A at Line 6: Mysterious character "~".
```

这说明词法分析器在接入 Bison 后仍然保留了实践 1.1 的扩展能力，并且能够继续扫描后续输入，报告多个词法错误。

### 3. 输出格式分析

对于无错误输入，语法树输出采用先序遍历。非终结符节点输出节点名称和行号，例如 `Program (1)`；词法节点只输出 token 名称，对于 `ID`、`TYPE`、`INT`、`FLOAT` 额外输出词素或数值，例如 `ID: inc`、`TYPE: int`、`INT: 83`、`FLOAT: 0.000105`。子节点相对父节点缩进两个空格，符合课程要求。

对于有错误输入，程序不会输出任何语法树节点，只输出错误信息。这一点很重要，因为错误恢复过程中构造出的局部树可能不完整，如果继续打印会干扰评测结果。

### 4. 后续衔接分析

本实验生成的 AST 已经可以作为后续语义分析的输入。下一阶段可以在 AST 上实现符号表和类型系统，例如遍历 `ExtDef` 处理全局定义，遍历 `Def` 处理局部变量声明，遍历 `Exp` 进行类型推导和左值检查。由于 token 的行号和文本已经保存在 AST 节点中，语义错误报告可以直接从节点获取出错行号，而不需要重新读取源文件。

当前实验还没有检查语义错误。例如 `required_4_tree_struct.cmm` 中使用了未定义变量 `y`，这在本阶段不会报错，而是正常输出语法树。这符合课程安排，因为未定义变量、类型不匹配、函数参数错误等属于后续语义分析任务。

总体来看，本次实验完成了从词法分析到语法分析的衔接：词法阶段提供结构化 token，语法阶段构造 AST，错误处理区分 A/B 类型，输出格式满足实践要求，并为后续语义分析、中间代码生成和目标代码生成保留了清晰接口。
