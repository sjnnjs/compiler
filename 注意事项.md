# 编译原理实验通用注意事项

## 环境

- 默认使用 Ubuntu 20.04、GCC/G++ 7.5.0、Flex 2.6.4、Bison 3.5.1。
- 每次提交前至少运行对应实验目录下的 `make clean && make && make test`。
- `lex.yy.c`、`parser.tab.c`、`parser.tab.h` 和可执行文件属于生成物，除非老师明确要求，否则不提交。

## Git

- 每个实验阶段单独提交，提交信息写清楚阶段，例如 `1.2`、`2.0`。
- 不提交本地工具包、下载的 `.deb`、运行时缓存和无关压缩包。
- push 前用 `git status --short --ignored` 检查暂存范围。

## 代码

- 不把标准输出文本当作阶段接口。词法到语法传 token，语法到语义传 AST，语义到中间代码传通过检查的 AST 和符号/类型查询接口。
- 错误类型要严格匹配实验文档。说明文字可以不同，但错误类型和行号要尽量准确。
- 遇到一个本质错误后要尽量抑制连锁噪声，尤其是未定义变量、未定义字段、错误类型继续参与运算的情况。
- 不依赖 glib 等额外库，优先使用 C 标准库、Flex 和 Bison。

## 报告

- 报告重点写实现了哪些功能、如何编译、核心设计亮点和测试结果。
- 不大段粘贴代码，更多解释符号表、类型系统、AST 接口和错误恢复设计。
- 每个实验的样例独立放在对应实验目录的测试文件夹中，文件名体现错误类型或样例目的。

## 后续衔接

- 语义分析通过后再进入中间代码生成；有语义错误时不生成 IR。
- 符号表和类型系统要保留查询接口，避免中间代码阶段重新分析 AST 细节。
- 新增阶段时同步更新 `使用与接口设计说明.md`。
